# Stakeholder-Friendly 체크리스트 — 모든 stakeholder를 위한 산출물

> 산출물의 독자는 아키텍트가 아니다. CX·UX·PM·QA·Client/Server 개발자·기획자·임원 등 **모든 stakeholder**가
> 정렬·합의·실행할 수 있어야 한다. 요약(정보 손실)이 아니라 **쉽게 정리**(완전하되 접근 가능)가 목표다.
> ("비전문가/임원도 이해" 같은 낮춤 표현은 쓰지 않는다 — 누구를 낮춰 부르지 않는다.)

## 모든 공유 산출물 공통
- [ ] **한눈에** 절: 한국어로 설계 의도 / 선택 이유 / 기대 효과 / 영향 범위 / 주요 고려사항을 먼저.
- [ ] **한국어 우선**: 서술은 한국어, 기술 용어만 영어(Quality Attribute, Latency Budget 등).
- [ ] **도면·흐름은 mermaid**: 구조/흐름/배치는 줄글이 아니라 다이어그램 + 한 줄 intent caption.
- [ ] **줄글 덩어리 금지**: 긴 산문 대신 표·리스트·다이어그램으로 구조화.
- [ ] **용어(Glossary)**: 약어·전문용어를 쉬운 말로 푼 표.
- [ ] **숫자는 맥락과 함께**: 성능 값은 budget/QS 링크와 "좋다/나쁘다"의 기준을 같이.
- [ ] **결론 먼저**: 권고/결정/다음 액션을 앞에. 근거는 뒤에.
- [ ] **누가 무엇을**: 실행 담당(CX/UX/PM/Client/Server/QA/...)이 드러난다.

## 이해도 테스트 (모든 산출물이 답해야 함)
1. 왜 필요한가?
2. 안 하면 어떤 문제가 생기는가?
3. 어떤 선택을 했는가?
4. 왜 다른 선택을 하지 않았는가?
5. 누가 무엇을 해야 하는가?

## 산출물별
- architecture-brief: 한눈에 + Context 다이어그램 + 용어 + 결정 요약 + Stakeholder Responsibilities.
- as-is-intake: 한눈에 + as-is 모듈맵(mermaid) + 용어 + 미확인 가정 표.
- source record: 한국어 + 설계 영향(요구/제약/위험/가정/확인질문/아키텍처 영향).
- candidate / comparison: 후보별 동작/개발 view(mermaid) + 쉬운 trade-off 한 줄.
- evaluation / handoff: 한눈에 + 판정 표 + 리스크 쉬운 설명.

## 자가 점검 질문
- 이 도메인을 모르는 동료가 **다이어그램만 보고** 큰 그림을 잡을 수 있는가?
- 약어를 처음 보는 사람이 용어표로 막힘없이 읽히는가?
- 핵심 메시지가 **첫 화면 안**에 있는가?
- 읽고 나면 "내가(또는 우리 팀이) 무엇을 해야 하는지" 분명한가?
