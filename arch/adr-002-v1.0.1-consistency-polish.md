# ADR-002 — v1.0.1 업무 적용 전 일관성 개선 baseline 채택

- **Status:** Accepted
- **Date:** 2026-06-06
- **Owner Approval:** granted by owner request 2026-06-06
- **Drivers (QS):** (하네스 자체의 Consistency, Safety, Maintainability)

## Context
v1.0.0 seed는 구조와 기본 검증은 갖췄지만, sample 성능 수치가 실제 목표처럼 보이고, release ADR 정책과 검증기
동작이 다르며, 적용 프로젝트 산출물 경로와 scoped 분석 lifecycle이 충분히 명확하지 않았다.

## Decision drivers
- 미승인 성능 수치를 제품 목표로 오인하지 않게 하는 Safety.
- 문서 정책과 `archdev check` 동작을 일치시키는 Consistency.
- 대형/회사 저장소를 수정하지 않고 분석·공유하는 Scoped safety.
- STATUS/QS/ADR/view/evaluation drift를 차단하는 Maintainability.

## Considered options
| 옵션 | Safety | Consistency | 유지보수성 | 변경 범위 | 비고 |
|---|---|---|---|---|---|
| **A. v1.0.1 focused consistency polish** | 높음 | 높음 | 높음 | 중 | 채택 |
| B. 문서만 정리하고 검증기는 유지 | 중 | 낮음 | 낮음 | 낮음 | 정책/도구 불일치 지속 |
| C. 대규모 CLI·구조 재설계 | 중 | 중 | 낮음 | 높음 | 현재 목표 범위 초과 |

## Decision brief
- **Recommended:** A. 기존 방향을 유지하면서 placeholder, canonical path, lifecycle, sidecar, Mermaid sync,
  strict release ADR 검사를 추가한다.
- **Why:** 업무 적용 전 오인과 drift 위험을 줄이면서 production code나 대규모 CLI를 도입하지 않는다.
- **Trade-offs accepted:** 문서·템플릿 수가 늘어나지만 lifecycle matrix와 index로 탐색 비용을 제한한다.

## Consequences
- 최상위 `quality/`는 하네스 가이드이며 제품 승인 목표는 `arch/quality/`에만 둔다.
- self-hosted release에는 current version을 포함한 Accepted ADR이 필수다.
- 적용 프로젝트는 canonical `arch/` 경로를 사용하고, host repo 분석은 read-only scoped sidecar를 기본으로 한다.
- `archdev check`가 필수 lifecycle/sidecar 파일, sample target, generic plan, release ADR을 가볍게 검사한다.

## Non-Goals
- production architecture 작성
- 대규모 CLI 또는 외부 dependency 추가
- 실제 latency/memory target 확정
